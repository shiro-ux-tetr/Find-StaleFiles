# Find-StaleFiles

長期間更新されていないファイルを抽出し、経過年数ごとに CSV へ出力する PowerShell スクリプトです。

「ディスク容量が逼迫しているが、どれを消していいか分からない」という状況で、
**削除の判断材料をつくる**ことを目的にしています。

---

## 何をするか

指定フォルダ配下を走査し、最終更新日時 (`LastWriteTime`) を基準に
**1年以上 / 3年以上 / 5年以上** 更新されていないファイルを抽出します。

出力されるファイル:

| ファイル | 内容 |
|---|---|
| `StaleFiles_1y_*.csv` | 1年以上 3年未満 更新なし |
| `StaleFiles_3y_*.csv` | 3年以上 5年未満 更新なし |
| `StaleFiles_5y_*.csv` | 5年以上 更新なし |
| `StaleFiles_summary_*.log` | 指定フォルダ単位の集計ログ |

**3 つの CSV に重複はありません。**
1 つのファイルは、必ずいずれか 1 つの CSV にのみ出力されます。

区分は **実行日からの経過期間** で判定します（暦年ではありません）。
実行時に境界日を画面とログへ表示するので、そこで確認できます。

```
 区分の境界 (重複なし):
   1年CSV : 2023-07-12 より後 〜 2025-07-12 以前
   3年CSV : 2021-07-12 より後 〜 2023-07-12 以前
   5年CSV : 2021-07-12 以前
```

例えば 2026年7月12日に実行した場合、
2023年9月のファイルは経過 **2年9か月** であり、まだ 3年に達していないため
`1年CSV`（1年以上3年未満）に出力されます。

CSV の列:

| 列 | 用途 |
|---|---|
| ファイル名 | |
| 拡張子 | 種別の判断 |
| サイズ(KB) | 削除効果の判断 |
| 最終更新日時 | 判定の根拠 |
| 経過日数 | 実際に何日放置されているか |
| 階層 | 深い階層＝忘れられている可能性が高い |
| フォルダパス | 実際の場所 |

## 使い方

### かんたんに実行する（推奨）

`Run-FindStaleFiles.bat` を **ダブルクリック** してください。

PowerShell スクリプトは、既定の設定ではセキュリティ上の理由から実行できません
（`このシステムではスクリプトの実行が無効になっているため...` というエラーが出ます）。

このバッチは、**システムの設定を変更せずに**、この 1 回の実行だけ制限を回避して起動します。

### PowerShell から実行する

```powershell
# 走査 (出力はスクリプトと同じフォルダ)
.\Find-StaleFiles.ps1

# 対象と出力先を指定
.\Find-StaleFiles.ps1 -Path "D:\Data" -OutputDir "D:\Report"

# 複数の共有フォルダをまとめて走査
.\Find-StaleFiles.ps1 -Path "\\server\share\営業部", "\\server\share\開発部"
```

実行ポリシーで弾かれる場合は、次のいずれかで実行します。

```powershell
# 方法1: この実行だけ回避する（システムの設定は変わらない）
powershell -ExecutionPolicy Bypass -File .\Find-StaleFiles.ps1

# 方法2: 現在の PowerShell ウィンドウの間だけ許可する
#        ウィンドウを閉じれば元に戻る
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\Find-StaleFiles.ps1
```

バッチファイルにも引数を渡せます。

```
Run-FindStaleFiles.bat -Path "D:\Data"
```

**スクリプトはどこに置いても動作します。**
USB メモリ・共有フォルダ・デスクトップ、どこから実行しても結果は変わりません。

## 設定の変更

スクリプト冒頭の「設定」ブロックを編集してください。

### 走査対象フォルダ

`@( )` の中の行から、**行頭の `#` を外す**だけです。

```powershell
$DEFAULT_TARGET_PATHS = @(

    'D:\'
    'C:\Users\PC_User\OneDrive'
    '\\fileserver\share\XX部'

)
```

**カンマは不要です。** PowerShell の配列は改行が区切りとして機能します。
（カンマを付けても動きますが、付け忘れると意図しない結果になりエラーも出ないため、
このスクリプトでは改行区切りに統一しています。）

空のままにすると、ユーザープロファイル配下を走査します。

### その他の設定

```powershell
# ---- 出力先フォルダ ----
#   空文字ならスクリプトと同じ場所へ出力。
#   優先順位: 引数 -OutputDir > この定数 > スクリプトの場所
$DEFAULT_OUTPUT_DIR = ''          # 例) 'D:\Report'  '\\fileserver\share\StaleFiles'

# ---- 出力ファイル名 ----
#   {0} には実行日時 (yyyyMMdd_HHmmss) が入ります。
#   {0} を削除すると、実行のたびに同じ名前で上書きされます。
$CSV_NAME_1Y = 'StaleFiles_1y_{0}.csv'
$CSV_NAME_3Y = 'StaleFiles_3y_{0}.csv'
$CSV_NAME_5Y = 'StaleFiles_5y_{0}.csv'
$LOG_NAME    = 'StaleFiles_summary_{0}.log'

# ---- 並列数 ----
#   同時に走査するフォルダ数の上限。
$MAX_PARALLEL = 3

# ---- バッファサイズ ----
#   大量のファイルを扱う場合は 32KB / 64KB へ増やすと速くなることがあります。
$BUFFER_SIZE = 8KB
```

定数で運用を固定しつつ、必要なときだけ引数で一時的に上書きできます。

## 設計上の判断

### 1. 列挙エンジンに robocopy を使う

`Get-ChildItem -Recurse` は書きやすい一方、ファイル数が多い環境では処理時間が問題になります。
そこで **robocopy のリストモード (`/L`)** を列挙エンジンとして利用しました。

```
robocopy <対象> NULL /L /S /NJH /NJS /FP /NDL /NC /BYTES /TS /R:0 /W:0
```

- `/L` … コピーせず一覧のみ出力する
- `/FP` … フルパスで出力する
- `/BYTES` `/TS` … サイズと更新日時を取得する
- `/R:0` `/W:0` … アクセスできないファイルでリトライして停止しないようにする

robocopy は Windows 標準コマンドのため、**追加インストールが不要**という利点もあります。

### 2. robocopy が折り返したパスを復元する

robocopy の出力は、**パスが長いと行の途中で折り返されます**。
そのまま解析すると、深い階層のファイルは **パスが途中で切れた状態** で CSV に出力され、
そのパスからファイルを開けなくなります。

対策として、出力行を次のように判別しています。

- **サイズ・日時を持つ行** → 新しいレコードの開始
- **サイズ・日時を持たない行** → 直前のパスの続き（折り返し）

後者を検出したら、`Repair-TruncatedPath` で直前のパスに連結して復元します。
robocopy は桁揃えのために空白を挿入するため、連結時に断片の前後の空白を除去しています。

これは実際に走査してみるまで気づけない挙動で、**ここを処理しないとツールとして成立しません。**

### 3. RunspacePool で複数フォルダを並列に走査する

トップフォルダ単位で走査を分割し、**最大 3 フォルダを同時に走査**します。

`Start-Job` は 1 ジョブごとに PowerShell のプロセスを起動するため、
数十フォルダを回すには **起動コストが大きすぎます**。
`RunspacePool` は同一プロセス内でスレッドを使い回すため軽量です。

並列数の上限を 3 にしているのは、**増やせば速くなるとは限らない**ためです。
走査の大半はディスクの読み取り待ちであり、
ディスクの性能が上限に達すると、並列数を増やしても待ち行列が伸びるだけになります。

### 4. 並列書き込みのために writer を同期ラッパで包む

**`StreamWriter` はスレッドセーフではありません。**

複数のワーカーが同じ CSV へ同時に書き込むと、
行が混ざる・欠落する・例外になる、といった問題が起きます。
しかも **毎回は再現しない** ため、原因の特定が非常に難しくなります。

そこで、実体の `StreamWriter` を `TextWriter.Synchronized()` で包み、
ワーカーにはその **同期ラッパ** を渡しています。

```powershell
$raw = New-Object System.IO.StreamWriter($path, $false, $encoding, $BUFFER_SIZE)
$writers[$y] = [System.IO.TextWriter]::Synchronized($raw)   # これをワーカーへ渡す
```

書き込みは自動的に排他制御されます。
最後に閉じるときは、**ラッパではなく実体 (`$raw`) を閉じます**。

### 5. 3 つの CSV を開いたまま並行して書き込む

経過年数ごとに走査をやり直すと、**走査コストが 3 倍**になります。

そこで 3 つの `StreamWriter` を最初に開き、**処理が終わるまで閉じずに保持**して、
1 回の走査で該当する CSV へ振り分けて書き込みます。

### 6. バッファに 8KB 溜めてから書き出す

`AutoFlush = $false` とし、`StreamWriter` のバッファサイズを 8KB に設定しています。

1 行ごとにディスクへ書き出すと、**I/O が処理全体のボトルネック**になります。
バッファに溜めてまとめて書き出すことで、書き込み回数を抑えています。

同じ理由から、集計ログも開いたままバッファリングし、処理終了時にまとめて確定させます。

### 7. `finally` で必ずフラッシュして閉じる

バッファリングしている以上、**閉じ忘れるとバッファに残ったデータが失われます**。

例外が発生した場合でも書き出しが行われるよう、`try` / `finally` で
`Flush()` → `Close()` → `Dispose()` を確実に実行しています。

### 8. `ForEach-Object` の子スコープ対策にハッシュテーブルを使う

`ForEach-Object` のスクリプトブロックは子スコープで実行されるため、
中で変数に再代入しても、親スコープには反映されません。

そのため、走査中の状態（集計値・保留中のレコード）は
**ハッシュテーブルのメンバとして保持**し、変数の再代入ではなく
中身の書き換えとして扱うことで、確実に反映されるようにしています。

### 9. カレントディレクトリではなく、スクリプト自身の位置を基準にする

出力先の基準に **カレントディレクトリ** を使うと、
「どこから実行したか」で出力先が変わってしまいます。

```powershell
PS C:\Users\me> D:\tools\Find-StaleFiles.ps1
# カレントは C:\Users\me だが、スクリプトは D:\tools にある
```

この状態では、ツールを共有フォルダや USB メモリに置いて配布したとき、
利用者ごとに出力先が変わり、「出力されたファイルが見つからない」という
問い合わせにつながります。

そのため `$PSScriptRoot` を用いて **スクリプト自身が置かれている場所** を基準にし、
どこに置いても、どこから実行しても同じ動作になるようにしました。

`$PSScriptRoot` は PowerShell 3.0 以降で利用できます。
取得できない環境や実行方法に備えて、`$MyInvocation` からの取得、
さらにカレントディレクトリへのフォールバックを段階的に用意しています。

### 10. 設定用の定数名を、引数名と衝突しないようにする

PowerShell の **変数名は大文字・小文字を区別しません**。

そのため、設定用の定数を `$OUTPUT_DIR` という名前にすると、
パラメータの `$OutputDir` と **同一の変数として扱われます**。

```powershell
param([string] $OutputDir)   # 引数で受け取る
$OUTPUT_DIR = ''             # ← 同じ変数を空文字で上書きしてしまう
```

この状態では、引数で出力先を指定しても、直後の定数定義で消えてしまいます。
定数を書いた瞬間に引数が機能しなくなる、原因の分かりにくいバグになります。

そのため定数名を `$DEFAULT_OUTPUT_DIR` / `$DEFAULT_TARGET_PATH` とし、
引数名と衝突しないようにしました。

「引数 > 定数 > 既定値」という優先順位を明示することで、
定数で運用を固定しつつ、必要なときだけ引数で上書きできる構成にしています。

### 11. 実行ポリシーは、スクリプト側では解決できない

PowerShell の実行ポリシーに弾かれた場合、
**スクリプトは 1 行も実行されていません**。

そのため、スクリプトの中に「ポリシーを検出して自動的に回避する処理」を
書くことはできません。起動する前に止められているためです。

解決は **起動する側** で行う必要があります。

```bat
powershell.exe -NoProfile -ExecutionPolicy Bypass -File "%~dp0Find-StaleFiles.ps1" %*
```

`Run-FindStaleFiles.bat` はこれを行うだけの起動用バッチです。

重要なのは、**`Set-ExecutionPolicy` でシステムの設定を書き換えていない**ことです。
ツールを動かすために利用者のマシンのセキュリティ設定を恒久的に変更するのは、
影響範囲が大きすぎます。

`-ExecutionPolicy Bypass` は **その 1 プロセスにのみ適用**され、
システムの設定には一切影響しません。

なお `%~dp0` は「このバッチが置かれているフォルダ」を指します。
カレントディレクトリではないため、どこから起動しても
同じフォルダのスクリプトを確実に呼び出せます。

### 12. スクリプト本体は BOM 付き UTF-8 で保存する

**Windows PowerShell 5.1 は、BOM のない UTF-8 ファイルを
ANSI (日本語環境では cp932) として読み込みます。**

そのため、日本語のコメントや文字列を含むスクリプトを BOM なし UTF-8 で保存すると、
文字が化けた状態で解釈され、次のような **構文エラー** になります。

```
文字列に終端記号 " がありません。
ステートメント ブロックまたは型定義に終わりの '}' が存在しません。
```

化けた文字がたまたま引用符や波括弧として解釈されることで、
**コードの構造そのものが壊れる**ためです。
エラーメッセージが構文の問題を指すため、原因が文字コードだと気づきにくいのが厄介な点です。

これを避けるため、スクリプト本体は **BOM 付き UTF-8** で保存しています。

なお、PowerShell 7 以降は BOM なし UTF-8 を正しく読むため、この問題は起きません。
バージョンによって挙動が変わる点に注意が必要です。

### 13. バッチファイルは Shift-JIS (cp932) で保存する

日本語版 Windows のコマンドプロンプトは、既定の文字コードが **cp932 (Shift-JIS)** です。

バッチファイルを UTF-8 で保存すると、
**日本語のメッセージが文字化けします**。

そのため `Run-FindStaleFiles.bat` は cp932 で保存しています。
（PowerShell スクリプト本体は UTF-8 で問題ありません）

### 14. 到達できないフォルダがあっても、走査を止めない

走査対象に共有フォルダを指定する場合、
ネットワークの状態や権限によって **アクセスできないことがあります**。

1つでも到達できない時点で処理全体を停止する作りにすると、
「他のフォルダは走査できたはずなのに、何も出力されない」という
最も困る結果になります。

そのため、到達できないパスは **スキップして処理を継続** し、
スキップした事実を画面とログの両方に必ず残します。

```
 以下は走査できないためスキップします:
   - \\fileserver\share\旧営業部  [パスが存在しないか、アクセスできません]
```

**黙って結果から除外するのが最も危険**です。
利用者が「これで全部だ」と誤解したまま、削除の判断をしてしまうためです。

### 15. 削除機能をあえて実装していない

このツールはファイルの削除・移動を **一切行いません**。

自動削除の実装自体は難しくありませんが、
そのファイルが本当に不要かどうかを判断できるのは人間だけです。
更新日が古くても、年次で参照する重要な資料かもしれません。

**誤って削除した場合の損失は、ディスク容量の逼迫より遥かに大きい**と判断し、
出力を「判断材料の提供」に限定しました。

### 16. CSV を UTF-8 (BOM 付き) で出力する

BOM なしの UTF-8 で出力すると、**Excel で開いた際に日本語が文字化けします**。
CSV は Excel で開かれる前提のため、BOM 付きで出力しています。

## 動作環境

- Windows 10 / 11
- Windows PowerShell 5.1 以降（追加インストール不要）
- robocopy（Windows 標準コマンド）

**管理者権限は不要です。**

このツールが行うのはファイルの一覧取得と CSV の書き出しだけで、
必要なのは対象フォルダへの **読み取り権限** のみです。
ローカルの管理者権限とは別物のため、通常のユーザーのまま実行できます。

（読み取れないフォルダがあった場合は、スキップして処理を継続し、
その事実をログに記録します。）
